# Usability

## Description

As a non-technical backoffice user, I need the interface to be simple and efficient so that common administrative tasks can be completed with minimal training.

## Acceptance criteria

- A trained backoffice campus worker can create a student record, update a journey stage, log an interaction, create a task, and open a case without requiring technical terminology.
- The primary student record view presents current journey stage, assigned owner, active tasks, open cases, active alerts, and recent interaction history without additional navigation.
- Navigation labels use the same business terms as the requirements and feature specifications, including student record, journey stage, task, case, interaction, and risk flag.
- Validation messages identify the missing or invalid field and describe how the user can correct it.
- Common operational tasks do not require unnecessary duplicate data entry across student, task, case, and interaction workflows.
- First-release usability validation includes walkthrough testing with representative backoffice users before sign-off.
